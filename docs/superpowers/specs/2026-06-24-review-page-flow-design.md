# 标书核对独立页面与匹配流程设计

## 目标

将解析核对工作台从标书解析列表页拆分为独立页面，并明确“先核对解析结果，再主动匹配资质”的两阶段流程。

## 页面职责

### 标书解析页 `/tenders`

- 上传标书。
- 查看标书列表、解析状态和基础元数据。
- 已解析标书显示要求数量和确认进度。
- 提供“核对结果”按钮，跳转 `/tenders/:id/review`。
- 不在该页内展示完整核对工作台。

### 核对页 `/tenders/:id/review`

- 加载单个标书信息和要求列表。
- 展示左右分栏解析核对工作台。
- 提供返回标书列表。
- 用户修改、确认要求后主动点击“开始比对自身资质”。
- 存在未确认要求时弹出提示，允许返回继续核对或强制开始匹配。

### 匹配页 `/match/:id`

- 页面进入时只读取已有匹配结果，不自动触发匹配。
- 没有结果时显示“尚未匹配”及“开始匹配”按钮。
- 有结果时显示结果及“重新匹配”按钮。
- 只有显式点击按钮才调用匹配接口。

## 路由

- 新增 `/tenders/:id/review`。
- 保留 `/tenders`、`/match`、`/match/:id`。

## 验收标准

- 标书解析页不再嵌入核对工作台。
- 点击“核对结果”进入独立核对页面。
- 核对页可返回列表、继续编辑确认并进入匹配。
- 进入匹配页不会自动产生 API 匹配请求。
- 用户必须点击“开始匹配”或“重新匹配”才触发。
