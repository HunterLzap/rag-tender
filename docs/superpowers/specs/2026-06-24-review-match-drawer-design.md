# 核对工作台匹配与原文抽屉设计

## 目标

将标书核对流程调整为“先看完整解析列表，再手动匹配，最后集中处理异常”，避免列表与 PDF 长期并排导致两侧空间不足。

## 页面流程

1. 标书解析完成后，从标书列表点击“核对结果”进入独立核对页。
2. 页面默认只显示全宽、紧凑的解析结果列表，不加载 PDF 阅读器，不自动匹配知识库。
3. 用户点击“与知识库资质匹配”后启动后台匹配，页面停留在当前核对工作台。
4. 匹配完成后，原表追加“我的资质”和“匹配状态”列，并将不匹配、待人工核对等异常项优先展示。
5. 用户点击任意条目的“查看原文”或页码，右侧抽屉打开 PDF，并跳转至对应页；关闭抽屉后恢复完整列表视野。

## 数据与状态

- 解析状态沿用 `pending / confirmed / location_failed`。
- 匹配状态使用 `matched / unmatched / needs_review`。
- 前端统一将后端 `status` 映射为 `match_status`。
- 匹配结果通过 `requirement_id` 关联解析项，通过 `qualification_id` 关联知识库资质。
- 修改、新增或删除解析项后，当前匹配结果标记为失效并清空展示，要求用户重新匹配。
- 启动新匹配前同步清除旧匹配结果，避免轮询读取到上一轮数据。

## 交互与视觉

- 匹配前筛选解析状态；匹配后筛选匹配状态。
- 异常优先级：不匹配、待人工核对、未生成结果、已匹配。
- PDF 抽屉宽度约为视口的 46%，小屏占满宽度。
- 原文定位失败时仍可打开 PDF 首页，并明确显示“页码定位失败”。
- 保留批量确认、批量删除、行内编辑和手动新增。

## 验证

- 单元测试覆盖匹配字段归一化、异常排序和筛选。
- 后端测试覆盖触发新匹配前清除旧结果。
- 运行后端回归、前端 TypeScript 编译与生产构建。
- 浏览器验证进入核对页不产生匹配 POST，PDF 初始不展示，点击原文后以抽屉出现。
