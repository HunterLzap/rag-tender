# BidMaster-Pro 可学习点与落地任务

> 目的：记录对 `guangshu100/BidMaster-Pro` 的对比结论，明确哪些设计值得吸收，哪些重型架构暂不照搬，以及本项目后续可执行任务。
>
> 对比日期：2026-07-03
> 参考仓库：https://github.com/guangshu100/BidMaster-Pro

---

## 一、结论

BidMaster-Pro 更像“全流程招投标平台骨架”，范围覆盖招标解读、投标生成、投标检查、文档排版、资讯监控、权限管理、桌面客户端和 Docker 部署。

本项目更像“本地单人使用的标书解析 + 资质库 + 保守匹配核验工具”，核心优势是贴近真实资质核验：字段级证据矩阵、保守判定、错例沉淀、红线待办、规则库治理。

因此后续策略不是迁移到 BidMaster-Pro，也不是照搬它的平台架构，而是：

**学习它的模块化、检查项覆盖、流程表达和平台能力设计；保留本项目现有的字段级证据核验作为核心竞争力。**

---

## 二、最值得学习的设计

### 1. Skill 插件化机制

BidMaster-Pro 把能力拆成独立 Skill，例如资质检查、保证金检查、签章检查、文档排版、内容生成等。

本项目可学习点：

- 把红线规则、资质核验、技术参数响应、规则建议等能力逐步拆成可注册的检查器。
- 每个检查器有统一输入、统一输出、风险等级、证据项、建议动作。
- 新增检查项时尽量不修改主流程，只新增检查器并注册。

适合落地到：

- `submission_checklist_service`
- `match_service`
- `technical_response_service`
- `rule_library_service`

### 2. 全流程阶段设计

BidMaster-Pro 用“招标解读 / 投标生成 / 投标检查 / 文档排版”表达完整工作流。

本项目可以形成更清晰的本地工作流：

1. 标书解析
2. 人工核对
3. 资质库/业绩库准备
4. 保守匹配
5. 投标待办清单
6. 自动填写
7. 导出检查报告

当前项目已有前 6 步的基础能力，但用户路径和状态汇总还可以更明确。

### 3. 检查项覆盖面

BidMaster-Pro 的检查项覆盖范围更广，尤其是投标提交和废标风险类检查。

本项目可优先吸收这些检查：

- 授权委托书、法人身份证、被授权人身份证。
- 签字、签章、盖章、密封、封装。
- 正本、副本、份数、目录、页码、装订。
- 投标保证金金额、形式、截止时间、到账风险。
- 投标有效期是否满足招标要求。
- 报价是否超过预算、最高限价或控制价。
- 暗标中是否出现公司名、联系人、电话、地址等敏感信息。
- 电子投标文件命名、格式、上传截止时间、CA 签章。

### 4. LLM Gateway 能力

BidMaster-Pro 有统一 LLM Gateway 的思路。

本项目已经有 LLM / Embedding / Vision 三类配置，但还可以补：

- 模型调用重试。
- fallback 模型。
- 每次任务记录实际模型、耗时、token 或字符用量。
- 按任务类型选择默认模型，例如解析、结构化、复核、生成可使用不同模型。
- API 调用失败时给用户更明确的“可重试 / 可切换模型 / 可人工处理”提示。

### 5. 项目化数据模型

BidMaster-Pro 有 `Project` 概念，把文档、分析、章节、检查报告等归到一个项目下。

本项目目前以 `Tender` 为核心，后续可以升级为“投标项目”：

- 一个项目包含一份或多份招标文件。
- 一个项目绑定资质匹配结果、业绩匹配结果、投标待办、模板、导出文件、人工修正记录。
- 这样更符合真实工作习惯，也方便以后支持历史复盘和报告导出。

### 6. 桌面端包装

BidMaster-Pro 使用 Electron 做桌面客户端。

本项目当前 Web 本地版已经够用，桌面端不是近期刚需，但可以作为后期体验优化方向：

- 双击启动。
- 内置服务状态检测。
- 本地文件打开体验更自然。
- 降低用户理解“后端 + 前端端口”的成本。

## 三、不建议照搬的部分

### 1. 不建议短期引入 PostgreSQL / Redis / MinIO / Celery

本项目定位是本地单人使用，SQLite + 本地文件 + 后台任务已经足够。

引入重型基础设施会增加部署和维护成本，短期不会显著提升资质核验准确率。

### 2. 不建议照搬完整 Agent 框架

Agent Engine、LangGraph、多 Agent 编排适合复杂平台，但本项目当前最重要的是减少误判、提升证据链可信度。

过早引入 Agent 框架会增加调试复杂度，也可能让结论更不可控。

### 3. 不建议把字段级核验改回纯 LLM 判断

BidMaster-Pro 的一些检查更偏“把招标文本和投标文本交给 LLM 判断并返回 JSON”。

本项目现在的优势是：

- 相似度只做候选召回。
- 最终结论由字段级证据矩阵和规则判断。
- 缺证据默认 `needs_review`，不轻易自动通过。

这条路线应继续保留。

### 4. 不建议一次性扩成全流程投标生成平台

自动生成投标正文、AI 配图、完整排版系统属于更大产品范围。

本项目应先把“核标、查漏、证据、待办、自动填写”做稳，再考虑生成能力。

## 四、落地任务清单

### P0：近期优先做

| 任务 | 目标 | 说明 |
|:---|:---|:---|
| 抽象检查器接口 | 为规则/红线/资质核验模块化打基础 | 先不大改架构，可从红线检查和技术参数检查开始 |
| 扩展投标待办红线 | 降低废标风险 | 授权书、报价日期、暗标敏感信息、投标有效期、电子投标提交 |
| 检查报告导出 | 让匹配结果可交付 | 导出资质匹配结论、待确认项、缺失证据、红线待办 |
| LLM 调用记录 | 便于排错和复盘 | 记录模型、耗时、失败原因、任务类型，不记录明文 API Key |
| 补齐规则归并接口 | 闭环当前规则库风险 | `merge_custom_rule` 已被路由暴露但服务层缺实现 |

### P1：中期做

| 任务 | 目标 | 说明 |
|:---|:---|:---|
| 项目化模型 | 从“标书”升级为“投标项目” | 项目聚合标书、资质匹配、待办、模板、导出物 |
| 检查器注册表 | 支持按项目启停检查项 | 类似 Skill Registry，但保持轻量 |
| 模型 fallback | 提升可用性 | 主模型失败时可切换备用模型或提示人工处理 |
| 检查项版本化 | 规则变更可追溯 | 记录规则版本、启停、命中结果和生成时间 |
| 业绩匹配增强 | 提高业绩类自动判断质量 | 金额、时间、项目类型、甲方、验收状态结构化 |

### P2：后期考虑

| 任务 | 目标 | 说明 |
|:---|:---|:---|
| Electron 桌面端 | 改善本地使用体验 | 功能稳定后再包，不影响当前 Web 版 |
| Docker 部署 | 方便迁移和演示 | 只在需要跨机器部署时做 |
| 多用户/权限 | 支持团队协作 | 需要先明确是否从单人自用转团队使用 |
| 投标正文生成 | 扩展到生成场景 | 应建立在解析和核验稳定之后 |
| 全文一致性检查 | 检查投标文件内部矛盾 | 如金额、日期、公司名、项目名、人员名前后一致 |

## 五、建议的实现顺序

### 阶段 1：轻量插件化

目标：不引入复杂 Agent 框架，只把现有检查逻辑拆得更清楚。

建议做法：

1. 定义 `CheckContext`、`CheckResult`、`EvidenceItem`。
2. 把红线待办中的 5 类规则迁移为检查器。
3. 新增授权书、投标有效期、报价上限、暗标敏感信息检查器。
4. 前端仍展示在投标待办清单和匹配明细中。

### 阶段 2：报告化

目标：用户可以把系统结论拿去复核或交付。

建议输出：

- 标书基本信息。
- 资质匹配总览。
- 不通过项。
- 待确认项。
- 缺失证据。
- 红线待办。
- 人工修正记录。
- 规则命中说明。

### 阶段 3：项目化

目标：让系统从“处理一份标书”升级为“管理一个投标项目”。

数据关系建议：

```text
Project
  ├─ Tender
  ├─ TenderRequirement
  ├─ MatchResult
  ├─ SubmissionChecklist
  ├─ FillTemplate
  ├─ OutputDocument
  └─ MatchCorrection
```

### 阶段 4：桌面端或部署增强

目标：降低使用门槛。

可选方向：

- Electron 包装当前前端。
- 启动器负责后端进程、端口检测、日志查看。
- 不急于引入 PostgreSQL、Redis、MinIO。

---

## 六、保持不变的原则

1. 本地优先：默认单机、SQLite、本地文件，不强制云服务。
2. 证据优先：没有字段级证据，不自动判通过。
3. 保守判定：缺证据进入 `needs_review`。
4. 可追溯：匹配结果、人工修正、规则变更都要留痕。
5. 不泄露密钥：API Key 加密存储、脱敏展示、日志不输出明文。
6. 先稳后大：先把核验和待办做可靠，再扩生成、协作和平台化。

---

## 七、下一步推荐

近期最建议做的 5 件事：

1. 补齐 `merge_custom_rule`，让规则库当前暴露的归并接口闭环。
2. 设计轻量检查器接口，不引入复杂 Agent 框架。
3. 扩展投标待办红线：授权书、投标有效期、报价上限、暗标敏感信息。
4. 增加检查报告导出。
5. 为 LLM 调用增加任务级日志和失败复盘信息。
