# RAG-Tender Assistant 标书自动分析辅助系统 — 产品需求文档（PRD）

| 项目信息 | |
|:---|:---|
| **项目名称** | rag_tender_assistant |
| **文档版本** | v1.1 |
| **语言** | 简体中文 |
| **技术栈** | 后端 FastAPI + 前端 React + MUI + Tailwind CSS |
| **核心依赖** | RAG-Anything（二开）、MinerU、LightRAG、DeepSeek、硅基流动 BGE-M3 |

## 一、产品目标

### 1.1 一句话定位

**面向招投标从业者的本地化标书智能分析工具，自动完成标书解析、资质匹配与模板填写。**

### 1.2 核心价值主张

- **省时**：将标书人工研读 + 资质比对 + 填表的传统流程从数小时压缩至分钟级。
- **防漏**：通过结构化提取 + 语义匹配，确保标书中的资质要求"条条有着落"，不漏项。
- **可控**：全流程本地运行，资质文件不出本机；API 供应商可随时切换，不绑定单一平台。

### 1.3 成功指标（可量化）

| 指标 | 目标值 | 说明 |
|:---|:---|:---|
| 标书资质要求提取覆盖率 | ≥ 95% | 标书中明确的资质/业绩/财务/人员要求，系统应能提取出 ≥95% 的条目 |
| 资质匹配准确率 | ≥ 90% | 对"符合/不符合"的判定，人工抽检正确率 ≥90% |
| 单份标书解析耗时 | ≤ 3 分钟 | 100 页以内 PDF 标书，从上传到结构化输出 ≤3min |
| 自动填写覆盖率 | ≥ 80% | 标书模板中可由资质库直接填写的字段占比 ≥80% |
| 本地启动到可用 | ≤ 30 秒 | 从启动应用到界面可操作的时间 |

## 二、用户故事

| # | 角色 | 故事 | 价值 |
|:---:|:---|:---|:---|
| US-1 | 招投标从业者 | 我希望上传一份标书 PDF/Word 文件，系统自动解析出其中的资质要求、业绩要求、财务要求、人员要求等结构化信息 | 省去逐页人工研读标书的时间，快速掌握投标门槛 |
| US-2 | 招投标从业者 | 我希望查看解析后的标书要求列表，按"资质/业绩/财务/人员/其他"分类浏览，并可手动修正 | 确认解析准确性，纠正 OCR 或 LLM 的误判 |
| US-3 | 招投标从业者 | 我希望有一个"知识库区"，把营业执照、ISO 证书、3C 证书、业绩合同、人员证书、财务报表等文件统一上传管理 | 集中管理公司资质资产，一处维护、多处复用 |
| US-4 | 招招投标从业者 | 我希望系统自动解析知识库中的证书/文档，提取出证书名称、编号、有效期、发证机构等关键信息 | 免去手工录入资质信息，且过期时能及时发现 |
| US-5 | 招投标从业者 | 我希望系统自动将标书的资质要求与我知识库中的资质进行匹配，给出"符合/不符合"结论及原因 | 快速判断投标资格，避免投了标才发现资质不够 |
| US-6 | 招投标从业者 | 我希望系统能根据匹配结果，自动填写标书投标文件模板中对应的内容（如证书编号、有效期、企业信息等） | 减少投标文件编制中的重复性填表工作 |
| US-7 | 招招投标从业者 | 我希望能在运行时通过弹窗切换 LLM / Embedding / Vision 的 API 供应商和模型，并测试连接是否正常 | 灵活适配不同供应商，避免被单一 API 锁定 |

## 三、需求池

### P0 — 必须做（核心流程）

| 需求ID | 需求名称 | 优先级 | 描述 | 验收标准 |
|:---|:---|:---:|:---|:---|
| P0-01 | 标书上传与解析 | P0 | 支持 PDF/DOCX/DOC 文件上传，通过 RAG-Anything + MinerU 解析为结构化内容（文本、表格、图片）。**包含附件解析**：标书中的附件（技术规格书、图纸等）一并解析提取 | ① 上传 PDF 后能返回结构化 JSON；② 能正确识别标题层级；③ 能提取表格数据；④ 附件内容被解析并标记来源；⑤ 100 页文档 ≤3min |
| P0-02 | 资质要求提取 | P0 | LLM 从解析后的标书内容中提取资质/业绩/财务/人员要求，分类结构化输出 | ① 输出按 5 类（资质/业绩/财务/人员/其他）分类；② 每条含"要求描述+类型+是否硬性"；③ 覆盖率 ≥95% |
| P0-03 | 资质要求展示 | P0 | 前端分类展示提取的标书要求列表，支持展开/折叠、查看原文定位 | ① 按分类 Tab 切换；② 每条可跳转到标书原文对应位置；③ 支持搜索筛选 |
| P0-04 | 知识库文件上传 | P0 | 知识库区支持上传全格式文件：PDF / 图片（PNG/JPG）/ DOCX / 扫描件（自动 OCR）/ XLSX（财务报表），按类型分类（企业资质/人员资质/业绩/财务） | ① 支持拖拽上传；② 上传时选择分类；③ 文件列表显示名称/类型/上传时间/状态；④ 扫描件自动触发 OCR；⑤ XLSX 文件可通过 openpyxl 解析 |
| P0-05 | 知识库自动解析与资质提取 | P0 | 系统自动解析知识库中的文件，LLM/Vision 从证书图片、文档中提取资质信息。**提取字段（7 项）**：证书名称、编号、有效期、发证机构、认证范围、等级、持证主体 | ① 上传后自动触发解析；② 解析状态实时更新（待处理/解析中/完成/失败）；③ 解析结果包含 7 项字段；④ 解析结果可查看和编辑 |
| P0-06 | 资质信息管理 | P0 | 知识库中的资质信息可查看、编辑、删除、手动新增 | ① 资质列表展示提取结果；② 支持编辑各字段；③ 支持手动新增资质条目；④ 支持删除 |
| P0-07 | 匹配引擎 | P0 | 将标书要求与知识库资质进行匹配，结合语义检索（向量相似度）+ 规则校验。**规则校验支持 5 类数值**：注册资本金额、合同金额、营业收入、资产负债率、人员年限。**对每个不符合项输出**：①不符合的具体点 ②期望的资质类型/参数 ③建议用户检查知识库是否已上传该资质（防止"有资质但忘传文件"） | ① 每条标书要求给出"符合/不符合/需人工确认"；② 符合的关联到具体资质条目；③ 不符合的给出①具体不符合点 ②期望资质 ③知识库检查建议；④ 支持 5 类数值规则自动校验 |
| P0-08 | 匹配结果展示 | P0 | 前端以对照表形式展示标书要求 vs 我的资质，通过/不通过状态一目了然。**不符合项额外展示**：期望资质（需要什么样的资质）+ 知识库状态（是否已上传该资质文件，防止漏传） | ① 双栏对照（左：标书要求，右：匹配资质）；② 颜色标识通过/不通过；③ 不符合项可展开查看：具体不符合点 + 期望资质 + 知识库是否已有；④ 可展开查看匹配详情和原因 |
| P0-09 | 自动填写 | P0 | 根据匹配结果，将符合条件的资质信息自动填入用户上传的投标文件模板。**支持多格式模板输入**（DOCX/PDF/XLSX），自动识别格式后填写。**输出格式**：DOCX + PDF 双输出（PDF 通过 LibreOffice 转换） | ① 支持上传 DOCX/PDF/XLSX 模板；② 自动识别模板格式；③ 能识别模板中的占位/字段；④ 填写后可下载 DOCX 和 PDF；⑤ 不符合的字段标注并跳过 |
| P0-10 | API 配置弹窗 | P0 | 运行时弹窗配置三组 API（LLM / Embedding / Vision），每组含 base_url、api_key、model_name，支持供应商预设 + 自定义，含测试连接，持久化存储 | ① 弹窗含三组独立配置卡片；② 每组有供应商下拉预设（DeepSeek/硅基流动/智谱等）+ 自定义选项；③ 每组有"测试连接"按钮并返回结果；④ 配置持久化到本地（SQLite/配置文件）；⑤ API Key 在界面脱敏显示（仅显示末 4 位），不写入日志 |
| P0-11 | 本地部署 | P0 | 系统可在 Windows 本地一键启动（后端 + 前端），无需云服务 | ① 提供 Windows 启动脚本；② SQLite 本地存储；③ 启动后自动打开浏览器访问前端 |

## 四、UI 设计稿

### 4.0 配色规范

> **全局配色：主色调淡紫色 + 白色背景**
>
| 用途 | 色值 | 说明 |
|:---|:---|:---|
| 主色（Primary） | `#7C4DFF` 或 `#8B5CF6` | 淡紫色，用于导航栏选中态、主按钮、强调元素 |
| 主色浅（Primary Light） | `#EDE7F6` 或 `#F3F0FF` | 极淡紫，用于 hover 背景、选中行背景 |
| 背景（Background） | `#FFFFFF` | 白色主背景 |
| 次背景（Surface） | `#F9F7FF` | 带淡紫色调的浅灰，用于卡片背景、内容区 |
| 成功（Success） | `#4CAF50` | 绿色，符合状态 |
| 警告（Warning） | `#FF9800` | 橙色，需确认/即将过期 |
| 错误（Error） | `#EF5350` | 红色，不符合状态 |
| 文字主色 | `#333333` | 深灰，正文 |
| 文字次色 | `#666666` | 中灰，辅助说明 |

> 以下所有线框图的配色均遵循上述规范：白色背景 + 淡紫色主色调。

## 六、技术约束与假设

| 约束/假设 | 说明 |
|:---|:---|
| 本地单机部署 | Windows 10/11，SQLite 存储，无多用户/权限/云服务 |
| RAG-Anything 二开 | 基于 MIT License 的 RAG-Anything，MinerU 做 PDF/Office 结构提取 |
| 大模型外部调用 | LLM/Embedding/Vision 均通过 API 调用外部服务，需用户自行配置 API Key |
| 文档转换依赖 | Office 文档 → PDF 依赖 LibreOffice 26.2 本地安装；DOCX/PDF 双输出也依赖 LibreOffice |
| 扫描件 OCR | 扫描件通过 MinerU OCR 能力处理，需确保 MinerU OCR 模型已正确安装 |
| XLSX 解析 | 财务报表等 XLSX 文件通过 openpyxl 解析 |
| 多格式模板填写 | 自动填写支持 DOCX（python-docx）/ PDF（pdfplumber+填写）/ XLSX（openpyxl）三种模板格式 |
| UI 配色 | 固定淡紫色(#7C4DFF) + 白色配色方案，不支持深色模式 |
| 网络要求 | 解析和匹配过程需联网调用 API；知识库存储和前端操作离线可用 |
| 性能假设 | 标书 ≤100 页为常见场景；超大标书（>200页）可能需分块处理 |
