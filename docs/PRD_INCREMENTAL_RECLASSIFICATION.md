# RAG-Tender Assistant 标书要求分类改造 — 增量 PRD

| 项目信息 | |
|:---|:---|
| **文档类型** | 增量 PRD（基于 PRD.md v1.1 扩展） |
| **改造主题** | 标书要求按"处理动作"重新分类：企业资质 / 产品技术参数 / 投标待办提交件 |
| **版本** | v1.2-incr |
| **日期** | 2026-06-25 |
| **依据** | `docs/REQUIREMENT_RECLASSIFICATION_PLAN.md`（改造宪法） |
| **技术栈** | 后端 FastAPI + 前端 React + MUI + Tailwind CSS + SQLite（不变） |
| **部署** | 本地 Windows 单机，单人自用（不变） |

> **原始需求复述**：标书解析出的"资质"项里混入了设备参数要求和提交件要求，导致匹配引擎对所有要求一视同仁去找企业资质，产出大量无意义的 unmatched 结果。本次改造按"后续处理动作"将要求拆为三类（企业资质类 / 产品技术参数类 / 投标待办提交件类），并为业绩/财务/人员三类增加 capability/submission 子维度，使匹配引擎只对 capability 类走匹配，产品参数走技术响应表，提交件走待办清单。

---

## 一、产品目标

| # | 目标 | 衡量标准 |
|:---:|:---|:---|
| G1 | **消除匹配噪音**：设备参数、提交件不再进入企业资质匹配流程，匹配结果不再出现无意义的 unmatched | 重新解析样本标书后，"资质匹配结果"中 0 条设备参数类、0 条纯提交件类要求 |
| G2 | **产品参数有归属**：设备/产品技术参数有独立的技术响应表，可逐条标注响应状态 | 技术响应页展示所有 product_spec 类要求，每条有响应状态字段 |
| G3 | **投标待办可追踪**：提交件类要求汇聚为 checklist，可管理准备状态 | 待办清单页展示所有 submission 类要求，每条有状态字段 |
| G4 | **能力 vs 提交可区分**：业绩/财务/人员三类内部区分 capability（走匹配）与 submission（走待办） | 这三类每条要求带 requirement_nature 标识，匹配引擎只处理 capability |

## 三、需求池

### P0 — 必须做（核心分类与分流）

| 需求ID | 需求名称 | 描述 | 验收标准 |
|:---|:---|:---|:---|
| P0-R1 | 三类分类落地 | 重写 `tender_service.py` 的 `_EXTRACT_SYSTEM_PROMPT`，将标书要求按处理动作分为三类：①企业资质类（qualification，保留）②产品技术参数类（product_spec，新增）③投标待办提交件类（submission，新增）。更新 `REQUIREMENT_CATEGORIES` 枚举。从原 qualification 中移除"设备证明/检测报告参数"和"制造商授权书(作提交件)" | ①LLM 能将"输出功率≥200kW"归入 product_spec；②LLM 能将"提交法人授权委托书"归入 submission；③LLM 能将"营业执照""ISO认证"归入 qualification；④样本标书重新解析后分类合理 |
| P0-R2 | 子维度字段落地 | 为每条要求增加 `requirement_nature` 字段，取值 `capability`（能力要求，走匹配）/ `submission`（提交材料，走待办）。LLM 在提取时自动判断并输出该字段。针对 performance/financial/personnel 三类生效 | ①"近3年类似业绩≥3个"→ nature=capability；②"提交业绩证明材料"→ nature=submission；③"注册资本≥500万"→ nature=capability；④"提交财审报告"→ nature=submission；⑤qualification/product_spec 类默认 nature=capability |
| P0-R3 | 匹配引擎分流 | 改造 `match_service.py` 的 `match_tender()`：只对 `requirement_nature=capability` 的要求走现有匹配流程（语义检索+5类数值规则+LLM判断）；`product_spec` 类跳过匹配；`submission` 类跳过匹配。匹配结果不再出现 product_spec/submission 类要求 | ①匹配结果中 0 条 product_spec；②匹配结果中 0 条 nature=submission；③现有 5 类数值规则对 capability 类仍正常工作；④回归测试通过 |
| P0-R4 | 数据模型扩展 | `TenderRequirement` 模型新增 `requirement_nature` 字段（可选，默认 capability，向后兼容）；`category` 枚举新增 `product_spec`、`submission`。数据库迁移脚本配套提供 | ①新字段可读写；②旧数据 nature 默认 capability 不报错；③数据库迁移可执行 |

### P1 — 重要（新页面与体验）

| 需求ID | 需求名称 | 描述 | 验收标准 |
|:---|:---|:---|:---|
| P1-R1 | 技术响应页 | 新增前端页面 `/tenders/{id}/technical`，展示所有 product_spec 类要求的技术响应表。每条含：参数要求描述、响应状态（满足/偏离/优于/待响应，默认待响应）、实际值（用户可手填）、备注。支持按响应状态筛选 | ①页面列出所有 product_spec 要求；②每条可编辑响应状态和实际值；③支持筛选；④响应状态有颜色标识 |
| P1-R2 | 待办清单页 | 新增前端页面 `/tenders/{id}/checklist`，展示所有 submission 类要求（含各类的 submission 子维度）的提交件 checklist。每条含：材料名称、来源类别（资质/业绩/财务/人员/产品提交件）、状态（未开始/进行中/已完成，默认未开始）、备注（用于灰色地带提示）。**不设截止时间字段、不设责任人字段**（用户确认 Q4） | ①页面列出所有 submission 要求；②每条可编辑状态；③状态有颜色标识（未开始灰/进行中橙/已完成绿）；④支持按状态筛选 |
| P1-R3 | 匹配结果页改造 | 匹配结果页只展示 capability 类匹配结果；移除 product_spec/submission 类的"未匹配资质"噪音。顶部概要栏数字只统计 capability 类 | ①匹配结果页无设备参数、无纯提交件；②概要栏数字准确反映 capability 类匹配情况；③现有符合/不符合/需确认展示不变 |
| P1-R4 | 分类筛选与导航调整 | 标书详情/核对页的分类标签从 5 个调整为按三大组组织：企业资质（qualification/performance-capability/financial-capability/personnel-capability/other-capability）、产品技术参数（product_spec）、投标待办（各 submission）。匹配结果/技术响应/待办清单可通过 Tab 或入口切换 | ①分类标签清晰反映三类；②可在三组间切换浏览；③导航不混乱 |

### P2 — 可选（迁移与边界）

| 需求ID | 需求名称 | 描述 | 验收标准 |
|:---|:---|:---|:---|
| P2-R1 | 历史数据重新解析 | 为已解析的历史标书提供"按新分类重新解析"按钮，重新触发 LLM 提取（用新 prompt），覆盖旧的 category 和 requirement_nature | ①历史标书可一键重新解析；②重新解析后分类按新规则；③解析中状态可见；④不丢失用户手动编辑的匹配确认 |

## 五、对 6 个待确认问题的产品决策（已用户确认 ✅）

| # | 问题 | 最终决策 | 用户确认 |
|:---:|:---|:---|:---:|
| Q1 | 历史数据迁移：重新解析按钮 vs 迁移脚本？ | **提供"按新分类重新解析"按钮**。LLM 重新分类比规则脚本准确；单人自用、量小、成本可控；重新解析只覆盖 category 和 requirement_nature，保留用户手动确认的匹配结果。列为 P2-R1。 | ✅ 已确认 |
| Q2 | 子维度自动化：LLM 自动判断 vs 手动调整？ | **LLM 自动判断 + 允许手动调整**。与现有 P1-01（解析结果人工校对）一致，LLM 先判、用户可改。manual 调整只改 requirement_nature 字段，不触发重新匹配（除非用户主动点"重新匹配"）。 | ❌ 无需再问，沿用现有校对机制 |
| Q3 | 技术响应表"实际值"来源：手填 vs 产品库匹配 vs 只列要求？ | **P1 阶段只列要求 + 用户手填实际值**。当前无产品库功能，引入产品库匹配是独立大特性，超出本次范围；单人自用，手填即可满足编制技术响应文件需求。产品库匹配可作为未来 P2 特性。 | ❌ 无需再问，先做最简方案 |
| Q4 | 待办清单状态机：哪些状态？是否要责任人/截止时间？ | **状态三档：未开始/进行中/已完成。不设截止时间字段，不设责任人字段**。单人自用，责任人均为自己；截止时间多余，三档状态就够。 | ✅ 已确认 |
| Q5 | 灰色地带（如"提供制造商授权书"兼具提交件+隐含资格）：单归 vs 双归？ | **单一归类，按主处理动作归 submission，备注栏标注隐含含义**。避免数据重复、状态管理复杂；主处理动作是"提交"，归 submission 进待办清单；备注标注"隐含代理资格"提示用户。列为 P2-R2。 | ✅ 已确认 |
| Q6 | 匹配结果页：三类分别入口 vs 统一页面 Tab 切换？ | **统一页面 Tab 切换**（企业资质匹配 / 技术响应表 / 投标待办清单）。同一标书的所有分析结果集中查看，减少导航跳转；Tab 切换符合单人多任务浏览习惯；与现有左侧导航结构兼容。 | ❌ 无需再问，Tab 方案最优 |

## 五·补充、对 5 个新增待确认问题的主理人决策

| # | 问题 | 主理人决策 | 依据 |
|:---:|:---|:---|:---|
| NQ1 | 一条要求兼具 capability 和 submission 时，拆成两条还是保留一条？ | **拆两条**。capability 条走匹配，submission 条走待办。LLM prompt 应引导拆分（如"近3年类似业绩≥3个，并提交业绩证明材料"→ 拆为 capability 条"业绩≥3个" + submission 条"提交业绩证明材料"） | 拆分更清晰，各走各的流程，避免单条 nature 取舍难题 |
| NQ2 | "other"类如何处理 nature 字段？ | **other 类也区分 capability/submission**。兜底但保留子维度，与其它类一致 | 一致性原则，避免 other 类成为"黑洞" |
| NQ3 | 技术响应表/待办清单是否需要与知识库联动？ | **本次不做联动**，保持独立 | 联动涉及知识库结构调整，超出本次范围。列为未来特性 |
| NQ4 | 重新解析时，category 变更是否影响已有匹配结果？ | **清空受影响条目的旧匹配结果**。capability 类保留匹配结果，product_spec/submission 类不保留 | 数据一致性，避免旧匹配结果与新分类矛盾 |
| NQ5 | LLM prompt 中是否需要为每类提供 few-shot 示例？ | **由架构师在 prompt 设计中决定**，建议加入 2-3 个 few-shot 示例，特别是 capability/submission 边界案例 | 这是 prompt 工程细节，架构师设计阶段定 |

| # | 问题 | 背景 | 建议倾向 |

| NQ1 | 一条要求兼具 capability 和 submission 时，拆成两条还是保留一条？ | 如"近3年类似业绩≥3个，并提交业绩证明材料"——前半是能力要求、后半是提交件。拆两条更清晰但增加条目数；保留一条则 nature 只能取一个 | 倾向**拆两条**：capability 条走匹配，submission 条走待办。LLM prompt 应引导拆分 |
| NQ2 | "other"类如何处理 nature 字段？ | 现有 other 类是兜底，改造后是否也区分 capability/submission？还是 other 类统一归 submission？ | 倾向 other 类也走 capability/submission 区分，兜底但保留子维度 |
| NQ3 | 技术响应表/待办清单是否需要与知识库联动？ | 如待办清单"已完成"状态是否自动关联已上传的知识库文件？技术响应"实际值"能否从知识库产品资料自动带出？ | 本次**不做联动**，保持独立。联动涉及知识库结构调整，超出范围。列为未来特性 |
| NQ4 | 重新解析时，category 变更是否影响已有匹配结果？ | 历史标书重新解析后，部分要求从 qualification 变为 product_spec/submission，其旧匹配结果如何处理？ | 倾向**清空受影响条目的旧匹配结果**，capability 类保留匹配结果，新增/变更类不保留 |
| NQ5 | LLM prompt 中是否需要为每类提供 few-shot 示例？ | 新分类比旧 5 类更细，LLM 可能需要示例才能准确区分 capability vs submission | 建议架构师在 prompt 设计中加入 2-3 个 few-shot 示例，特别是 capability/submission 边界案例 |
