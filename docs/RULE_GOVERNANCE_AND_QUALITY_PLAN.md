# 标书匹配质量治理与规则库规划

> 目的：把“标书解析、资质匹配、错例沉淀、规则治理、人工复核”的系统边界固定下来，避免后续靠临时需求堆功能。
>
> 当前原则：这是严肃的投标辅助系统，宁可保守进入人工复核，也不能把证据不足的要求自动判为符合。

---

## 1. 当前系统定位

本系统不是“相似度匹配工具”，而是“证据链核验 + 人工复核 + 规则沉淀”的投标辅助系统。

当前核心链路：

```text
标书解析
  ↓
要求分类：资质 / 业绩 / 财务 / 人员 / 技术响应 / 投标待办
  ↓
候选召回：关键词、同义词、字段文本、语义辅助
  ↓
字段级证据矩阵核验
  ↓
匹配结论：符合 / 不符合 / 需确认
  ↓
人工确认与纠错
  ↓
错例库
  ↓
规则建议 / 规则草案 / 自定义规则
  ↓
后续匹配更保守、更可解释
```

关键边界：

- 相似度只允许做候选召回，不能直接决定“符合”。
- 关键字段缺失、组合证据不足、规则未覆盖时，默认进入“需确认”。
- 自定义收紧规则只允许把“符合”降级为“需确认”，不能把“不符合/需确认”升级为“符合”。
- 人工确认不是直接修改算法，而是先沉淀错例，再通过审核流程变成规则。

## 2. 当前已经具备的能力

### 2.1 匹配质量控制

- 字段级证据矩阵：
  - 证书类型
  - 持证主体/持证人员
  - 有效期
  - 适用范围
  - 等级
  - 纳税证明 + 社保证明组合要求
- 业绩匹配收紧：
  - 有明确金额要求时，金额满足才可自动通过。
  - 泛泛“真实有效业绩”不能仅凭业绩库存在项目自动通过。
- 投标待办红线：
  - 投标保证金
  - 截止时间
  - 签章密封
  - 正副本、装订、页码、份数
  - 样品、演示、踏勘、答辩

### 2.2 错例库

当前错例库记录人工确认产生的纠错记录：

- 系统原判断
- 人工最终确认
- 人工填写原因
- 标书要求
- 资质信息
- 当时的证据快照

当前错例库的定位是“事实记录层”，不是“自动学习层”。

### 2.3 规则库

当前规则库支持：

- 内置规则目录展示。
- 内置规则启用/停用。
- 启停原因记录。
- 规则变更日志。
- 从错例中生成规则建议。
- 规则建议接受/拒绝。
- 接受建议后生成规则草案。
- 规则草案发布/驳回。
- 发布后生成自定义收紧规则。
- 已发布启用的自定义收紧规则参与后续匹配。

当前规则建议只处理一种高风险情况：

```text
系统原判：符合
人工改判：不符合
```

这说明系统规则过松，需要生成“收紧规则”建议。

## 3. 当前不能假装已经完成的能力

以下能力目前没有完整实现，后续不能在交互或文档中说成“已完成”：

- 手动新增规则。
- 直接编辑已发布规则内容。
- 删除规则。
- 规则版本管理与回滚。
- 规则相似度检测与合并。
- 规则适用范围配置。
- 错例聚类统计。
- 基于错例的解析策略优化建议。
- “需确认 → 符合”的放宽规则建议。
- 规则命中后的效果评估。

---

## 4. 风险分级

### P0 风险：必须优先防住

这些风险会直接导致投标判断错误：

- 没有资质却显示符合。
- 证据字段缺失却显示符合。
- 只有部分组合证据却显示符合，例如只有纳税证明，没有社保证明。
- 规则停用后业务仍然命中。
- 自定义规则发布后没有参与后续匹配。
- 规则重复或冲突，导致解释混乱。

### P1 风险：需要尽快治理

- 错例记录越来越多，但没有聚合分析。
- 人工改判原因不规范，后续难以转规则。
- 规则只能新增，不能版本化管理。
- 规则命中后没有统计效果，无法判断规则是否有效。

### P2 风险：体验与效率问题

- 规则建议太多，审核成本高。
- 相似规则重复展示。
- 错例库只能列表查看，不能筛选、分组、导出。

## 5. 规则治理原则

### 5.1 新增规则原则

新增规则必须满足：

- 有明确业务风险。
- 有真实错例或真实标书条款支撑。
- 能解释命中原因。
- 有明确适用范围。
- 不直接把结果升级为“符合”。
- 默认先作为“收紧规则”或“复核规则”。

不允许新增这类规则：

- 过宽规则，例如“所有证书都必须人工复核”。
- 无证据来源规则，例如“感觉这个容易错”。
- 只靠一句模糊描述就影响所有匹配的规则。
- 与已有规则高度重复但没有合并说明的规则。

### 5.2 修改规则原则

修改规则必须保留历史：

- 不直接覆盖旧规则导致审计断层。
- 修改前后内容都要可追溯。
- 修改原因必须填写。
- 如果影响范围扩大，必须进入草案审核。
- 如果只是改名称、说明、错别字，可以走轻量变更。

### 5.3 删除规则原则

默认不物理删除规则。

推荐策略：

- 业务不用了：停用。
- 规则误伤严重：停用并记录原因。
- 规则被新规则替代：标记为“已替代”，记录替代规则 ID。
- 只有测试数据或误建草案可以允许删除。

### 5.4 相似规则处理原则

新增或发布规则前必须做相似检查：

```text
新规则
  ↓
查找相似内置规则 / 自定义规则 / 草案
  ↓
如果相似：
  - 合并到已有规则
  - 扩展已有规则适用范围
  - 替代旧规则
  - 保留为新规则但必须说明差异
```

相似判断维度：

- 要求标题相似。
- 关键词重合。
- 证据项重合。
- 适用分类相同。
- 命中动作相同。
- 人工纠错原因相似。

## 6. 规则影响边界

### 6.1 当前允许的影响

当前自定义规则只允许做一件事：

```text
原本自动符合
  ↓ 命中自定义收紧规则
改为需确认
```

这能防止系统过度自信，不会制造新的自动通过风险。

### 6.2 后续可以扩展的影响类型

| 规则类型 | 作用 | 风险等级 | 默认策略 |
|:---|:---|:---|:---|
| red_flag | 投标待办标红 | 中 | 可启停 |
| tighten_rule | 符合降级为需确认 | 低 | 优先支持 |
| required_evidence_rule | 强制要求某些证据项 | 中 | 需要审核 |
| exclusion_rule | 命中后直接不符合 | 高 | 谨慎支持 |
| relax_rule | 需确认升级为符合 | 极高 | 暂不自动生效 |
| parse_hint_rule | 优化解析提示词/字段提取 | 中 | 先做建议，不直接改结果 |

## 7. 错例库后续规划

错例库不应止步于“记录列表”，下一阶段应变成“质量分析台”。

### 7.1 错例分类

按系统原判和人工改判分类：

- `matched → unmatched`：系统过松，应生成收紧规则建议。
- `matched → needs_review`：系统过度自信，应生成复核规则建议。
- `needs_review → unmatched`：系统召回到候选但证据不足，可优化不符合判定。
- `needs_review → matched`：系统过保守，只能生成低置信优化建议，不能直接放宽。
- `unmatched → matched`：系统漏判，需检查资质解析、候选召回或同义词。

### 7.2 错例聚合

增加聚合维度：

- 按要求标题聚合。
- 按要求分类聚合。
- 按资质类型聚合。
- 按证据项聚合。
- 按人工原因关键词聚合。
- 按命中/未命中的规则聚合。

### 7.3 错例输出建议类型

错例分析不应只输出规则建议，还应区分：

- 规则建议：当前规则过松或缺少证据项。
- 解析建议：标书要求拆分不准确或漏提。
- 资质库建议：资质文件解析字段缺失。
- 同义词建议：候选召回没找到正确证据。
- 人工流程建议：需要用户补充证据文件。

## 8. 规则库后续规划

### 阶段 A：先补治理能力

目标：让规则库可控，而不是只会越堆越多。

需要实现：

- 规则详情页。
- 手动新增规则草案。
- 编辑草案。
- 发布前相似规则检查。
- 发布前影响预估。
- 已发布规则停用原因必填。
- 规则命中次数统计。

验收标准：

- 新增规则不会直接影响线上匹配，必须先进入草案。
- 发布规则前必须展示相似规则。
- 发布规则后重新匹配能看到命中提示。

### 阶段 B：错例分析台

目标：让错例库从“记录”变成“分析”。

需要实现：

- 错例分类统计。
- 高频错例榜。
- 同类错例聚合。
- 一键生成规则建议。
- 一键生成解析优化建议。
- 错例处理状态：未处理 / 已生成建议 / 已忽略 / 已解决。

验收标准：

- 用户能看到哪些问题重复发生。
- 用户能从一组错例生成一个规则建议，而不是一条错例生成一条规则。
- 已处理错例不会反复干扰审核。

### 阶段 C：规则版本与影响评估

目标：让规则变更可审计、可回滚、可评估。

需要实现：

- 规则版本表。
- 发布记录。
- 替代关系。
- 回滚到上一版本。
- 规则命中历史。
- 发布前试跑：选择一份标书，预览规则会影响哪些结果。

验收标准：

- 每条规则能看到历史版本。
- 每次修改都有原因和时间。
- 发布规则前能看到影响范围。

### 阶段 D：谨慎放宽机制

目标：处理“需确认 → 符合”的高频情况，但不能直接放松系统。

需要实现：

- 放宽建议单独分类。
- 放宽建议必须有多条同类错例支撑。
- 放宽规则默认只给提示，不自动改成符合。
- 若允许自动符合，必须要求完整证据项和人工明确开启。

验收标准：

- 放宽不会默认影响匹配结论。
- 用户必须能看到为什么系统建议放宽。
- 放宽规则可随时停用并回滚。

## 9. 后续实施顺序

推荐顺序：

1. 规则详情与草案编辑。
2. 相似规则检查。
3. 错例分类统计。
4. 高频错例聚合。
5. 从聚合错例生成规则建议。
6. 规则命中统计。
7. 发布前试跑影响预估。
8. 规则版本与回滚。
9. 放宽建议机制。

不建议现在直接做：

- 直接删除规则。
- 直接编辑已发布规则并立即生效。
- 自动学习后自动改规则。
- 自动把“需确认”升级成“符合”。

---

## 10. 开发执行纪律

后续每个功能必须按以下方式推进：

1. 先说明该功能属于哪个阶段。
2. 说明它解决哪个风险。
3. 说明会影响哪些数据表、API、页面、匹配结论。
4. 先写失败测试。
5. 实现最小闭环。
6. 跑相关后端测试。
7. 前端改动必须跑 `npm run build`。
8. 汇报本次完成内容、验证结果、剩余风险。

每次最多推进一个闭环，不再连续堆多个未验收功能。

## 11. 当前下一步建议

当前最应该做的是：

```text
阶段 A-1：规则详情与草案编辑
```

原因：

- 规则库已经有建议、草案、发布，但用户还不能认真查看和编辑草案内容。
- 如果没有详情和编辑能力，后续新增规则会变成不可控堆叠。
- 这一步不会直接扩大规则影响范围，风险较低。

下一步验收目标：

- 规则库可以打开某条规则/草案详情。
- 草案可以编辑名称、描述、关键词、适用分类、命中动作。
- 发布前显示“这是收紧规则，只会把符合降级为需确认”。
- 发布后后续匹配能命中并在匹配明细解释原因。
