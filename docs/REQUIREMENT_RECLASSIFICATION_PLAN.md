# 标书要求分类改造方案

> 📅 立项时间：2026-06-25
> 🎯 状态：方案已与用户对齐，待启动标准 SOP（PRD → 架构 → 实现 → QA）
> 📝 本文件是本次改造的"宪法"，后续 PRD/架构/实现都以本文件为准

---

## 一、改造背景

### 问题现象

用户在实测中发现，标书解析出来的部分"资质"项实际并不属于企业资质范畴：

1. **设备参数要求**（如"输出功率≥200kW"）被归到 `qualification`，拿去知识库匹配营业执照/ISO，永远 unmatched，纯浪费 LLM 调用
2. **提交件要求**（如"提交法人授权委托书"、"提交资格承诺书"）被归到 `qualification` 或 `other`，拿去匹配企业资质也无意义——它们不是企业"具备"的东西，而是投标时"要交"的东西

### 根因分析（代码定位）

| 位置 | 问题 |
|:---|:---|
| `backend/app/services/tender_service.py` 第 62-70 行 `_EXTRACT_SYSTEM_PROMPT` | `qualification` 类明确列了"能力证明：生产/供货能力、**设备证明**、检测报告"和"授权要求：**制造商授权书**、代理商资格" → 设备参数被引导塞进资质类的根源 |
| 同文件第 95-104 行 `other` 类 | 兜底"任何'投标人须知''资格性审查'章节下的任何条件" → 授权委托书、资格承诺书等提交件被兜进来 |
| `backend/app/services/match_service.py` `match_tender()` | 对 5 类要求一视同仁地走"语义检索知识库 + 数值规则 + LLM 判断"找企业资质，没有按类型分流 |

### 问题本质

当前 5 类分类（qualification/performance/financial/personnel/other）回答的是"这是关于什么的"（**主题维度**），但没回答"拿到这条要求之后系统该干嘛"（**处理动作维度**）。结果匹配引擎对所有要求一视同仁去找企业资质，自然产出无意义结果。

## 二、改造方向（已与用户对齐）

### 2.1 按"后续处理动作"拆成三类

| 类别 | 性质 | 后续该干嘛 | 当前状态 |
|:---|:---|:---|:---|
| **企业资质类** | 企业长期持有的凭证 | 匹配知识库资质（当前已支持） | ✅ 保留 |
| **产品技术参数类**（新增） | 投标产品须满足的规格 | 生成技术响应/偏离表，不碰知识库 | ❌ 新增 |
| **投标待办/提交件类**（新增） | 投标时需准备的文件 | 生成提交清单 checklist，不碰知识库 | ❌ 新增 |

### 2.2 业绩/财务/人员三类内部区分"能力要求"vs"提交材料"子维度

用户确认：这三类本身分类没问题，但内部要区分两个子维度：

- **能力要求**（capability）：企业"具备"的东西 → 走匹配（如"近3年类似业绩≥3个"）
- **提交材料**（submission）：投标时"要交"的东西 → 走待办清单（如"提交业绩证明材料"）

### 2.3 产品检测报告的归属（用户已确认）

按用户意见：
- **参数要求**（如"提供功率≥200kW 的设备"）→ 归**产品技术参数类**
- **提交报告动作**（如"提供第三方型式试验报告"）→ 归**投标待办/提交件类**

## 三、三类划分详细定义

### 3.1 企业资质类（qualification）

**定义**：企业作为投标主体长期持有的法定凭证和能力证明。

**典型举例**：
- 主体资格：营业执照、法人资格、独立承担民事责任能力
- 专业资质：ISO 认证、3C 认证、行业许可证、安全生产许可证、特种设备制造/安装许可证
- 资质证书：建筑资质（总承包/专业承包等级）、承装(修、试)电力设施许可证
- 人员资质（建造师证等）—— 注：人员类单独分类，但"持证"这件事算企业能力
- 信用记录：失信被执行人限制、重大税收违法记录
- 法律合规：参加政府采购活动前三年内无重大违法记录

**后续动作**：匹配知识库资质（语义匹配 + 5 类数值规则 + LLM 判断）

**注意**：从原 `qualification` 中**移除**以下内容（迁到新类）：
- ❌ "设备证明、检测报告"（参数部分）→ 迁到产品技术参数类
- ❌ "制造商授权书"作为提交件 → 迁到投标待办类
- ❌ "生产/供货能力"作为陈述 → 保留在资质类（属于企业能力）

### 3.2 产品技术参数类（product_spec）— 新增

**定义**：投标产品本身须满足的技术规格、性能指标、工艺要求。

**典型举例**：
- 设备功率、电压等级、容量、转速等性能参数
- 材质要求（不锈钢/碳钢/铝合金等）
- 工艺要求（焊接标准、表面处理等）
- 性能指标（效率、精度、寿命等）
- 产品检测/试验的参数要求（如"通过 IP65 防护等级测试"）
- 技术规格偏离项

**后续动作**：
- 生成**技术响应表**（参数逐条响应：满足/偏离/优于）
- 不进知识库匹配
- 前端单独页面展示

**数据结构建议**：复用 `tender_requirements` 表，`category = 'product_spec'`，新增字段记录响应状态。

### 3.3 投标待办/提交件类（submission）— 新增

**定义**：投标过程中需要准备、签署、提交的文件和材料。

**典型举例**：
- 法人授权委托书
- 资格承诺书
- 中小企业声明函
- 投标/履约保证金
- 保密协议、反商业贿赂承诺
- 制造商授权书（作为提交件）
- 业绩证明材料（作为提交件，区别于"具备业绩"这个能力）
- 财审报告（作为提交件，区别于"财务指标达标"这个能力）
- 人员证书（作为提交件，区别于"持证人员"这个能力）
- 产品检测报告（作为提交件）

**后续动作**：
- 生成**提交清单 checklist**（材料名称 + 是否已准备 + 责任人 + 截止时间）
- 不进知识库匹配
- 前端单独页面展示

**数据结构建议**：新建 `submission_checklist` 表，或复用 `tender_requirements` 表 `category = 'submission'` + 新增 checklist 状态字段。

### 3.4 子维度区分（针对 performance/financial/personnel）

每条要求增加 `requirement_nature` 字段：

| 取值 | 含义 | 后续动作 |
|:---|:---|:---|
| `capability` | 能力要求（企业具备什么） | 走匹配引擎 |
| `submission` | 提交材料（投标要交什么） | 走待办清单 |

**举例**：
- "近3年类似业绩≥3个" → `category=performance, requirement_nature=capability` → 走匹配
- "提交近3年业绩证明材料" → `category=performance, requirement_nature=submission` → 走待办清单
- "注册资本≥500万" → `category=financial, requirement_nature=capability` → 走匹配
- "提交财审报告" → `category=financial, requirement_nature=submission` → 走待办清单

## 四、改造范围（用户确认：彻底改）

### 4.1 后端

| 模块 | 改动 |
|:---|:---|
| `tender_service.py` | 重写 `_EXTRACT_SYSTEM_PROMPT`：三类定义 + 子维度字段；更新 `REQUIREMENT_CATEGORIES` 枚举 |
| `tender_service.py` | LLM 输出 schema 增加 `requirement_nature` 字段 |
| `match_service.py` | `match_tender()` 改为分流：只对 `capability` 类走匹配，`product_spec` 走技术响应，`submission` 走待办清单 |
| 新增 `technical_response_service.py` | 处理产品技术参数的响应表生成 |
| 新增 `submission_checklist_service.py` | 处理待办清单生成与管理 |
| `models/tender.py` | `TenderRequirement` 增加 `requirement_nature` 字段 |
| `database.py` | 新增表或字段（待架构师定） |
| `api/tenders.py` / 新增 api | 暴露技术响应、待办清单的 API 端点 |

### 4.2 前端

| 模块 | 改动 |
|:---|:---|
| 现有标书详情/核对页 | 分类筛选标签从 5 个改为 7 个（资质/业绩/财务/人员/产品参数/待办/其他），或重新组织为"企业资质""产品技术""投标待办"三大组 |
| 新增 技术响应页 | `/tenders/{id}/technical` 展示产品参数响应表 |
| 新增 待办清单页 | `/tenders/{id}/checklist` 展示提交件 checklist |
| 匹配结果页 | 只展示 capability 类的匹配结果，product_spec/submission 不再出现"未匹配资质"的噪音 |
| 类型定义 `types/index.ts` | 新增 `requirement_nature`、`product_spec`/`submission` category 类型 |

### 4.3 数据库

- `tender_requirements` 表：新增 `requirement_nature` 字段（可选，向后兼容）
- 可能新增 `technical_responses` 表（产品参数响应）

## 六、待确认问题（PRD 阶段需产品经理厘清）

1. **历史数据迁移**：已解析的标书（如 148 页样本）如何处理？是否提供"按新分类重新解析"按钮，还是写迁移脚本把旧 category 重新归类？
2. **子维度自动化**：`requirement_nature` 由 LLM 自动判断，还是允许用户手动调整？
3. **技术响应表的数据来源**：产品参数的"实际值"从哪来？用户手填，还是从产品库匹配，还是只列要求让用户后续填？
4. **待办清单的状态机**：待办项有哪些状态（未开始/进行中/已完成/已上传）？是否需要责任人、截止时间字段？
5. **三类之间的边界灰色地带**：如"提供制造商授权书"——既是提交件又隐含代理资格。归一类还是双归？
6. **匹配结果页是否保留三类分别入口**：还是统一一个页面用 Tab 切换？
